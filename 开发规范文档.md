# 通义灵码开发提示词文档 - Vue/Nuxt 管理后台

**最后更新**: 2026-04-14  
**项目版本**: 1.0.0  
**Node.js**: >= 22.x (推荐 v22.22.2)  
**pnpm**: >= 9.x (推荐 v9.11.0)

本提示词文档用于指导通义灵码在生成 Vue/Nuxt 项目代码时遵循统一的技术栈、架构规范和编码风格。特别地，本文档明确了从 Element
Plus 迁移至 Nuxt UI 时如何保留原有界面布局与样式，确保迁移后的视觉一致性。

---

## 一、项目概述

本项目是一个基于 Nuxt 3+ 的现代化管理后台，采用 Vue 3 + TypeScript + Pinia + Nuxt UI 技术栈，遵循 **按业务特性组织（Feature-Sliced
Design 思想）** 的目录结构，并内置全栈开发能力（Nitro 服务端）。目标是构建一个可维护、可扩展、类型安全、完美支持 SSR 的中后台应用。

**迁移背景**：原项目基于 Element Plus 实现，现替换为 Nuxt UI 以解决 SSR 兼容问题，并提升开发体验。在迁移过程中，需尽量保留原
Element Plus 版本的整体布局、色彩、间距等视觉风格，仅将组件实现替换为 Nuxt UI 等效组件，并辅以 Tailwind 微调样式。

**关键说明：**

- `pages/` 仅做页面级组件组装，业务逻辑下放到 `composables`、`stores` 或 `modules` 中。
- `components/` 下的组件默认自动导入，无需手动 import。
- `composables/` 下的组合式函数也自动导入（以 `use` 开头）。
- `stores/` 下的 store 通过 Pinia 自动导入（已配置 `@pinia/nuxt`）。
- `modules/` 用于将紧密相关的业务代码（组件、组合式函数、store 切片）内聚在一起，每个模块通过 `index.ts` 暴露公共
  API，禁止跨模块直接引用内部文件。
- `plugins/` 目录下的插件自动加载，`.client.ts` 后缀表示仅在客户端运行（由于 Nuxt UI 原生支持 SSR，大部分情况下无需此标记）。
- `server/api/` 目录下的文件按 HTTP 方法命名：`[name].[method].ts`（如 `login.post.ts`）。
- `stores/types/` 存放共享的 TypeScript 类型定义，供多个 store 引用。
- `types/` 存放全局类型定义（如 `api.d.ts`），所有模块可共享。
- `.env` 文件存储环境变量（如后端 API 地址），不提交到 Git，参考 `.env.example`。
- **Nuxt UI 配置**：在 `nuxt.config.ts` 的 `modules` 数组中添加 `'@nuxt/ui'`，即可自动启用 Tailwind、图标模块和暗色模式支持。无需额外配置。

## 四、从 Element Plus 迁移至 Nuxt UI 的样式与布局保留指南

在迁移过程中，原 Element Plus 项目的布局结构、色彩风格、间距、圆角等视觉特征应尽量保留，以保证用户无感知切换。以下为迁移时的核心原则与操作指南。

### 1. 保留布局结构

原项目通常使用 `el-container`、`el-aside`、`el-main` 构建左右布局。Nuxt UI 不提供类似容器组件，需用 **Tailwind 布局类**
实现相同效果。

| 原 Element Plus 布局组件        | 迁移方案（Tailwind + Nuxt UI）                                       |
|----------------------------|----------------------------------------------------------------|
| `<el-container>`           | `<div class="flex h-screen">`                                  |
| `<el-aside width="200px">` | `<aside class="w-50 bg-gray-100 dark:bg-gray-800">` （宽度根据原值转换） |
| `<el-main>`                | `<main class="flex-1 overflow-auto p-4">`                      |

**示例：默认布局迁移**

```vue
<!-- 原 Element Plus 布局 -->
<template>
  <el-container class="layout-container">
    <el-aside width="200px">...</el-aside>
    <el-container>
      <el-header>...</el-header>
      <el-main>...</el-main>
    </el-container>
  </el-container>
</template>

<!-- 迁移后 Nuxt UI 布局 -->
<template>
  <div class="flex h-screen">
    <aside class="w-50 bg-gray-100 dark:bg-gray-800">...</aside>
    <div class="flex-1 flex flex-col">
      <header class="h-12 border-b">...</header>
      <main class="flex-1 overflow-auto p-4">...</main>
    </div>
  </div>
</template>
```

### 2. 保留色彩与主题

原 Element Plus 主题色（如 `#409EFF`）需通过 Tailwind 配置或 Nuxt UI 主题变量保持一致。

**方案 A**：在 `tailwind.config.js` 中扩展主题色，覆盖 Nuxt UI 默认的 primary 色。

```js
// tailwind.config.js
export default {
    theme: {
        extend: {
            colors: {
                primary: {
                    50: '#e6f7ff',
                    100: '#b3e0ff',
                    200: '#80c9ff',
                    300: '#4db2ff',
                    400: '#1a9bff',
                    500: '#409EFF',   // Element Plus 主色
                    600: '#0078d4',
                    700: '#005fa3',
                    800: '#00467a',
                    900: '#002d52',
                }
            }
        }
    }
}
```

**方案 B**：通过 Nuxt UI 的 `ui` 配置项全局修改组件颜色（更精细，但较复杂）。一般使用 Tailwind 扩展已足够。

### 3. 保留圆角、阴影与间距

Element Plus 组件默认圆角为 4px，阴影、间距等可通过全局 CSS 变量或 Tailwind 工具类统一设置。

在 `assets/css/main.css` 中覆盖 Nuxt UI 组件的圆角变量：

```css
:root {
    --ui-radius: 0.25rem; /* 4px */
}
```

对于原项目中使用自定义的间距（如 `margin: 20px`），应转换为 Tailwind 的 `p-4`、`m-5` 等，保持数值一致。

### 4. 组件样式替换对照

| Element Plus 组件 | Nuxt UI 等效组件                             | 样式保留技巧                                                             |
|-----------------|------------------------------------------|--------------------------------------------------------------------|
| `<el-button>`   | `<UButton>`                              | 使用 `color="primary"` 和 `variant="solid"` 模拟原样式；通过 Tailwind 调整圆角和阴影 |
| `<el-input>`    | `<UInput>`                               | 默认样式与 Element Plus 接近，可调整 `rounded` 和 `border-color`               |
| `<el-table>`    | `<UTable>`                               | 通过 `:ui` 属性定制表头背景、行高、边框等，与原表格风格对齐                                  |
| `<el-dialog>`   | `<UModal>`                               | 使用 `size="xl"` 等预设，配合 `:ui` 定制圆角和阴影                                |
| `<el-menu>`     | 自定义菜单（`<UNavigationMenu>` 或 Tailwind 实现） | 从原项目复制菜单样式类（如 `.el-menu-item`），用 Tailwind 重写                       |

**示例：按钮样式微调**

```vue

<UButton
    color="primary"
    variant="solid"
    class="rounded-md shadow-sm hover:shadow-md transition-shadow"
>
  提交
</UButton>
```

### 5. 图标迁移

原 Element Plus 使用 `@element-plus/icons-vue`，图标名称如 `User`、`Lock` 等。迁移到 Nuxt UI 后，需替换为 Heroicons 或
Lucide 图标名称，并修改为 `<UIcon>` 组件。

- 原代码：`<el-icon><User /></el-icon>`
- 新代码：`<UIcon name="i-heroicons-user-20-solid" />`

若想保持与原来一致的图标风格，可在项目中安装 `@iconify-json/heroicons` 并使用 outline 或 solid 版本。如需完全匹配，可自定义图标集。

### 6. 全局样式迁移

原 Element Plus 项目的全局样式（如 app.vue 中的背景、字体）需迁移到 `assets/css/main.css` 或 Nuxt UI 主题配置中。建议将原
CSS 文件中的样式逐一转换为 Tailwind 实用类或提取为组件样式。

## 五、编码规范

### 1. 命名约定

- **组件名**：PascalCase，如 `LoginForm.vue`、`UserDropdown.vue`
- **组合式函数**：以 `use` 开头，camelCase，如 `useMenu.ts`、`useUserStore`（但 store 本身是变量名）
- **Pinia store**：以 `use` 开头 + `Store` 结尾，camelCase，如 `useUserStore`、`useAppStore`
- **目录名**：kebab-case，如 `user-profile/`、`auth/`
- **接口/类型**：PascalCase，如 `User`、`MenuItem`、`LoginRequest`
- **API 路由文件**：`[name].[method].ts`，如 `login.post.ts`、`all.get.ts`
- **Store 文件名**：`[name].store.ts`，如 `user.store.ts`、`menu.store.ts`、`tasteVideo.store.ts`

### 2. 组件结构（Vue SFC）

严格按照以下顺序编写：

```vue

<script setup lang="ts">
  // 1. 类型导入（如果需要，但自动导入优先）
  // 2. props 和 emits 定义（使用 defineProps/defineEmits 泛型）
  // 3. 组合式函数调用（useStore、useRoute 等）
  // 4. 响应式数据定义
  // 5. 计算属性
  // 6. 方法/事件处理
  // 7. 生命周期钩子
</script>

<template>
  <!-- 模板内容，优先使用 Nuxt UI 组件，结合 Tailwind 辅助类 -->
</template>

<style scoped>
  /* 极少情况下使用，优先用 Nuxt UI 或 Tailwind 完成样式 */
</style>
```

### 3. Props & Emits 类型定义

使用 TypeScript 泛型 + withDefaults 定义 props：

```ts
const props = withDefaults(defineProps<{ title?: string }>(), {title: '默认标题'})
const emit = defineEmits<{ (e: 'update', id: number): void }>()
```

### 4. 自动导入规则

- Nuxt 自动导入 `components/`、`composables/`、`utils/` 下的文件，以及 `stores/`（需配置）。
- **禁止手动 import** 这些目录下的内容（除非类型定义或特殊场景）。
- Nuxt UI 组件和图标完全自动导入：无需手动 import，直接在模板中使用 `<UButton>`、`<UIcon name="i-heroicons-home" />`
  等。图标名称遵循 Iconify 命名约定（如 `i-heroicons-home`）。

## 六、状态管理规范（Pinia）

每个 store 独立文件，使用 `defineStore` 定义，推荐使用 Composition API 风格（函数形式）。

store 中可包含 state、getters、actions，actions 支持异步。

已在 `stores/types/` 目录下定义共享类型，避免重复定义。

**已实现的 Stores：**

- `user.store.ts`：用户信息、菜单、权限管理（user、menus、permissions、login、logout、fetchUserInfo）
- `app.store.ts`：应用 UI 状态（sidebarCollapsed、isFullscreen、theme）
- `performer.store.ts`：演员字典管理（performers、total、query、分页、查询、CRUD、批量删除）
- `tasteVideo.store.ts`：兴趣视频管理（list、total、query、performerDict、fetchList、createVideo、updateVideo、deleteVideo）

在 store 中使用封装的 `clientApiFetch`、`serverApiFetch` 调用 API（注意 SSR 上下文）。

**示例：**

```ts
// stores/user.store.ts
export const useUserStore = defineStore('user', () => {
    const user = ref<User | null>(null)
    const menus = ref<MenuItem[]>([])

    async function fetchUserInfo() {
        const data = await $fetch('/api/auth/me')
        user.value = data.user
        menus.value = data.menus
    }

    return {user, menus, fetchUserInfo}
})
```

## 七、数据获取与后台接口调用规范

### 1. 客户端数据获取

- **页面级数据**：使用 `useAsyncData` + `$fetch` 或直接 `useFetch`，可自动处理 SSR 水合和重复请求。
- **交互性请求**（如表单提交）：使用 `$fetch`（Nuxt 全局注入的 ofetch 实例）或封装的 `clientApiFetch`。
- **复杂缓存/后台更新**：引入 `@tanstack/vue-query`，并在插件中配置默认行为。
- **请求拦截/错误处理**：可通过插件配置 `$fetch` 的 `onRequest`、`onResponse` 拦截器，或通过 TanStack Query
  的 `defaultOptions`。
- **API 统一拦截**：`utils/api.ts` 中的 `interceptApiRequest` 会在客户端发起请求前检查登录状态。

### 2. 服务端 API（Nitro）

- API 路由位于 `server/api/` 目录，文件命名规范：`[name].[method].ts`，如 `login.post.ts`、`tree.get.ts`。
- 使用统一处理器 `defineApiEventHandler` 封装验证和守卫逻辑：
    - 自动合并 body 和 query 参数
    - Zod 运行时验证（失败返回 422）
    - 守卫函数数组（权限检查等）
    - 统一错误处理（H3Error 直接抛出，其他错误转为 500）
- 验证错误统一返回 422 状态码，业务错误按 REST 规范返回适当状态码。
- 调用真实后端 API 使用 `serverApiFetch()`，自动从 cookie 提取 token 并传递。
- Cookie 设置使用 HttpOnly（生产环境 HTTPS），防止 XSS 攻击。

```ts
// server/utils/defineApiEventHandler.ts
import {z} from 'zod'
import type {H3Event} from 'h3'

export function defineApiEventHandler<T extends z.ZodTypeAny>(config: {
    validation?: T  // Zod 验证 schema
    guards?: Array<(event: H3Event, payload: z.infer<T>) => Promise<void>>  // 守卫函数
    handler: (event: H3Event, payload: z.infer<T>) => Promise<any>  // 处理函数
}) {
    return defineEventHandler(async (event) => {
        // 1. 获取并合并参数（body + query）
        // 2. Zod 验证（失败返回 422）
        // 3. 执行守卫
        // 4. 执行处理函数
        // 5. 统一错误处理
    })
}
```

### 3. 类型安全

- 前后端共享类型：将类型定义放在 `stores/types/` 目录中（如 `menu.d.ts`、`tasteVideo.d.ts`）。
- API 响应使用 Zod 运行时校验，确保数据符合预期。
- 统一响应格式：

```ts
interface ApiResponse<T = any> {
    code: number      // 20000 表示成功
    data: T
    message: string
    success: boolean
    total?: number    // 列表接口可能有 total
}
```

- 错误处理：`code !== 20000` 时自动抛出错误，401 自动跳转登录页。
- 分页列表响应：

```ts
interface ListResponse<T = any> {
    list: T[]
    total: number
}
```

## 十、其他注意事项

- 所有生成代码需遵循 ESLint 和 Prettier 默认配置（已配置 eslint:recommended、plugin:
  vue/vue3-recommended、@typescript-eslint/recommended）。
- Nuxt UI 组件和图标完全自动导入，无需手动注册插件或单独安装图标包（默认包含
  Heroicons，如需其他图标集可安装对应的 `@iconify-json/xxx` 包）。
- **暗色模式**：Nuxt UI 通过 `@nuxtjs/color-mode` 内置支持，可通过 `useColorMode()` 切换模式，并在根元素自动添加 dark
  类。无需额外配置。若原 Element Plus 未使用暗色模式，可暂时忽略。
- 涉及路由跳转使用 `navigateTo`（Nuxt 3）而不是 `router.push`（但在组合式函数中可使用 `useRouter()`）。
- **SSR 兼容性**：Nuxt UI 原生支持 SSR，所有组件可在服务端渲染，无需特殊处理。
- **环境要求**：
    - Node.js >= 22.x（推荐 v22.22.2）
    - pnpm >= 9.x（推荐 v9.11.0）
    - 使用 `.npmrc` 配置依赖提升（shamefully-hoist=true、node-linker=hoisted）
- **依赖管理规范**：
    - **禁止直接安装 tailwindcss** - @nuxt/ui 已内置 Tailwind CSS 4，直接安装会导致版本冲突
    - **删除 tailwind.config.js** - Tailwind CSS 4 使用新架构，不需要配置文件
    - **CSS 导入语法** - 使用 `@import "tailwindcss";` 而非 `@tailwind base/components/utilities`
    - 若遇到 Tailwind 相关错误，优先检查是否存在上述违规配置
- **SSR 注意事项**（虽然 Nuxt UI 已兼容，仍需遵守 Nuxt SSR 最佳实践）：
    - 使用 `useState`、`useCookie` 等 Nuxt 工具保持状态同步
    - 使用 `useCookie` 读取/设置 cookie（SSR 安全）
    - 在 `onMounted` 或客户端钩子中才能访问 `window`、`document` 等浏览器 API
    - 使用 `import.meta.client` 和 `import.meta.server` 判断运行环境
- **性能优化**：

    - 生产环境移除 console 日志（vite.esbuild.drop）

## 十一、常见问题与解决方案

### 1. Tailwind CSS 样式不生效

**问题表现**：`bg-white`、`w-96` 等 Tailwind 工具类在页面中不生效

**可能原因与解决方案**：

#### 原因 A：使用了错误的 @tailwind 指令语法

- ❌ **错误**：`@tailwind base;` `@tailwind components;` `@tailwind utilities;`（Tailwind CSS 3 语法）
- ✅ **正确**：`@import "tailwindcss";`（Tailwind CSS 4 新语法）

**位置**：`assets/css/main.css` 文件开头

#### 原因 B：存在 tailwind.config.js 配置文件

- **问题**：Tailwind CSS 4 使用全新的基于 Rust 的 Oxide 引擎，不需要配置文件
- **解决**：删除项目根目录的 `tailwind.config.js`

#### 原因 C：直接安装了 tailwindcss 依赖

- **问题**：@nuxt/ui 已内置 Tailwind CSS 4.2.2，直接安装会导致版本冲突（常见为 3.4.x）
- **解决**：从 `package.json` 中移除 `tailwindcss` 依赖，重新执行 `pnpm install`

**正确的 package.json 依赖配置**：

```json
{
  "dependencies": {
    "@nuxt/ui": "^4.6.0"
    // ... 其他依赖
  },
  "devDependencies": {
    // 不应包含 tailwindcss
  }
}
```

### 2. colors.js default export 错误

**错误信息**：

```
Uncaught (in promise) SyntaxError: The requested module 
'/_nuxt/node_modules/tailwindcss/colors.js' does not provide an export named 'default'
```

**根本原因**：Tailwind CSS 版本冲突导致 CommonJS 导出格式不兼容

**解决方案**：

1. 从 `package.json` 移除直接安装的 `tailwindcss` 依赖
2. 删除 `tailwind.config.js` 配置文件
3. 确保 `main.css` 使用正确的 `@import "tailwindcss";` 语法
4. 清理缓存并重启：
   ```bash
   # 删除 .nuxt 和 node_modules/.vite 缓存
   if (Test-Path .nuxt) { Remove-Item -Recurse -Force .nuxt }
   if (Test-Path node_modules\.vite) { Remove-Item -Recurse -Force node_modules\.vite }
   
   # 重新启动
   pnpm dev
   ```

### 3. 构建文件缺失错误

**错误信息**：

```
Cannot find module '.\.nuxt\\dist\\server\\server.mjs' imported from .\.nuxt\dev\index.mjs
```

**原因**：构建缓存损坏或不完整

**解决**：完全清理 `.nuxt` 目录后重启开发服务器

### 4. Nuxt UI 图标警告

**警告信息**：

```
[Icon] Collection heroicons is not found locally
We suggest to install it via npm i -D @iconify-json/heroicons
```

**解决**：安装推荐的图标集

```bash
pnpm add -D @iconify-json/heroicons
```

# 通义灵码开发提示词文档 - Vue/Nuxt 管理后台

本提示词文档用于指导通义灵码在生成 Vue/Nuxt 项目代码时遵循统一的技术栈、架构规范和编码风格。请将此文档作为每次代码生成的上下文依据。

---

## 一、项目概述

本项目是一个基于 **Nuxt 3+** 的现代化管理后台，采用 **Vue 3 + TypeScript + Pinia + Element Plus** 技术栈，遵循 *
*按业务特性组织（Feature-Sliced Design 思想）** 的目录结构，并内置全栈开发能力（Nitro 服务端）。目标是构建一个可维护、可扩展、类型安全的中后台应用。

---

## 二、技术栈明细

| 层次          | 技术选型                                 | 说明                                                                                    |
|-------------|--------------------------------------|---------------------------------------------------------------------------------------|
| **元框架**     | Nuxt 3+                              | 基于 Vue 3、Vite、Nitro，支持 SSR/SSG、自动导入、文件路由、服务端 API                                      |
| **UI 组件库**  | Element Plus                         | 配合 `@element-plus/nuxt` 模块，图标使用 `@element-plus/icons-vue`                             |
| **样式方案**    | Tailwind CSS + Element Plus 主题       | Tailwind 用于布局和辅助类，Element Plus 提供组件样式，避免样式冲突                                          |
| **状态管理**    | Pinia                                | 使用 `defineStore` 定义 store，模块化存放于 `stores/` 目录，支持自动导入                                  |
| **客户端数据获取** | Nuxt 内置 `useFetch` / `$fetch`        | 页面级数据优先使用 `useAsyncData` + `useFetch`；交互性请求使用 `$fetch`；复杂缓存场景引入 `@tanstack/vue-query` |
| **服务端 API** | Nitro 事件处理器                          | 在 `server/api/` 中编写 API 路由；使用自定义 `defineApiEventHandler` 统一处理验证（Zod）和守卫               |
| **类型安全**    | TypeScript + Zod                     | 前后端共享类型（通过 `server/utils/types`）；运行时校验使用 Zod                                          |
| **构建工具**    | Vite (Nuxt 默认)                       | 极速开发体验                                                                                |
| **测试工具**    | Vitest + Vue Test Utils + Playwright | 单元测试和 E2E 测试（可选）                                                                      |

## 三、项目目录结构

采用按特性组织的方式，核心目录如下：

```
.
├── app.vue                     # 应用入口
├── nuxt.config.ts              # Nuxt 配置
├── pages/                      # 页面路由（文件即路由，只做组合，无业务逻辑）
├── layouts/                    # 布局组件
│   ├── default.vue             # 默认布局（含左侧菜单、顶栏）
│   └── guest.vue               # 访客布局（无菜单）
├── components/                  # 全局共享 UI 组件
│   ├── ui/                      # 通用基础组件（按钮、输入框等）
│   └── UserDropdown.vue         # 用户下拉菜单
├── composables/                  # 全局共享组合式函数
│   └── useMenu.ts                # 菜单处理（权限过滤、面包屑）
├── stores/                       # Pinia 状态模块
│   ├── user.store.ts              # 用户信息、菜单、权限
│   └── app.store.ts               # 应用 UI 状态（侧边栏折叠、全屏）
├── modules/                       # 核心业务特性模块（可选，适用于大型项目）
│   └── auth/                       # 认证特性
│       ├── components/              # 特性内私有组件（如 LoginForm）
│       └── index.ts                  # 公共出口
├── server/                         # 服务端 API 和中间件
│   ├── api/                         # API 路由
│   │   └── auth/                     # 认证相关接口
│   │       ├── login.post.ts
│   │       ├── logout.post.ts
│   │       └── me.get.ts
│   └── utils/                       # 服务端工具函数
│       └── defineApiEventHandler.ts  # API 统一处理器
├── middleware/                      # 路由中间件
│   └── auth.global.ts                # 全局登录守卫
├── public/                          # 静态资源
├── types/                           # 全局类型定义（可选）
└── utils/                           # 纯工具函数
```

**关键说明**：

- `pages/` 仅做页面级组件组装，业务逻辑下放到 `composables`、`stores` 或 `modules` 中。
- `components/` 下的组件默认自动导入，无需手动 import。
- `composables/` 下的组合式函数也自动导入（以 `use` 开头）。
- `stores/` 下的 store 文件通过 Pinia 自动导入（需配置 `@pinia/nuxt`）。
- `modules/` 用于将紧密相关的业务代码（组件、组合式函数、store 切片）内聚在一起，每个模块通过 `index.ts` 暴露公共
  API，禁止跨模块直接引用内部文件。

## 四、编码规范

### 1. 命名约定

- **组件名**：`PascalCase`，如 `LoginForm.vue`、`UserDropdown.vue`
- **组合式函数**：以 `use` 开头，`camelCase`，如 `useMenu.ts`、`useUserStore`（但 store 本身是变量名）
- **Pinia store**：以 `use` 开头 + `Store` 结尾，`camelCase`，如 `useUserStore`、`useAppStore`
- **目录名**：`kebab-case`，如 `user-profile/`、`auth/`
- **接口/类型**：`PascalCase`，如 `User`、`MenuItem`、`LoginRequest`

### 2. 组件结构（Vue SFC）

严格按照以下顺序编写：

```vue

<script setup lang="ts">
  // 1. 类型导入（如果需要，但自动导入优先）
  // 2. props 和 emits 定义（使用 defineProps/defineEmits 泛型）
  // 3. 组合式函数调用（useStore、useRoute 等）
  // 4. 响应式数据定义
  // 5. 计算属性
  // 6. 方法/事件处理
  // 7. 生命周期钩子
</script>

<template>
  <!-- 模板内容，优先使用 Element Plus 组件，结合 Tailwind 辅助类 -->
</template>

<style scoped>
  /* 极少情况下使用，优先用 Element Plus 或 Tailwind 完成样式 */
</style>
```

### 3. Props & Emits 类型定义

使用 TypeScript 泛型 + `withDefaults` 定义 props：

```ts
const props = withDefaults(defineProps<{ title?: string }>(), {title: '默认标题'})
const emit = defineEmits<{ (e: 'update', id: number): void }>()
```

### 4. 自动导入规则

- Nuxt 自动导入 `components/`、`composables/`、`utils/` 下的文件，以及 `stores/`（需配置）。
- **禁止手动 import** 这些目录下的内容（除非类型定义或特殊场景）。
- 对于 Element Plus 组件和图标，需手动导入（如 `import { Lock } from '@element-plus/icons-vue'`），因为图标未自动注册。

## 五、状态管理规范（Pinia）

- 每个 store 独立文件，使用 `defineStore` 定义，推荐使用 Composition API 风格（函数形式）。
- store 中可包含 state、getters、actions，actions 支持异步。
- 在 store 中直接使用 `$fetch` 调用 API（客户端）或 `useFetch`（但需注意 SSR 上下文）。
- 示例：

```ts
// stores/user.store.ts
export const useUserStore = defineStore('user', () => {
    const user = ref<User | null>(null)
    const menus = ref<MenuItem[]>([])

    async function fetchUserInfo() {
        const data = await $fetch('/api/auth/me')
        user.value = data.user
        menus.value = data.menus
    }

    return {user, menus, fetchUserInfo}
})
```

## 六、数据获取与后台接口调用规范

### 1. 客户端数据获取

- **页面级数据**：使用 `useAsyncData` + `$fetch` 或直接 `useFetch`，可自动处理 SSR 水合和重复请求。
- **交互性请求**（如表单提交）：使用 `$fetch`（Nuxt 全局注入的 ofetch 实例）。
- **复杂缓存/后台更新**：引入 `@tanstack/vue-query`，并在插件中配置默认行为。
- **请求拦截/错误处理**：可通过插件配置 `$fetch` 的 `onRequest`、`onResponse` 拦截器，或通过 TanStack Query
  的 `defaultOptions`。

### 2. 服务端 API（Nitro）

- API 路由位于 `server/api/` 目录，文件命名规范：`[method].[ext]`，如 `login.post.ts`。
- 推荐使用统一处理器 `defineApiEventHandler` 封装验证和守卫逻辑：

```ts
// server/utils/defineApiEventHandler.ts
import {z} from 'zod'

export function defineApiEventHandler<T extends z.ZodTypeAny>(config: {
    validation?: T
    guards?: Array<(event: H3Event, payload: z.infer<T>) => Promise<void>>
    handler: (event: H3Event, payload: z.infer<T>) => Promise<any>
}) {
    return defineEventHandler(async (event) => {
        // 合并参数、验证、守卫执行...
    })
}
```

- 验证错误统一返回 `422` 状态码，业务错误按 REST 规范返回适当状态码。

### 3. 类型安全

- 前后端共享类型：可将类型定义放在 `server/utils/types.ts` 中，或通过 monorepo 共享。
- API 响应使用 Zod 运行时校验，确保数据符合预期。

## 八、已实现的核心模块参考（可选）

为保持上下文一致，以下列出项目中已实现的关键模块，生成相关代码时可参考其命名和逻辑：

- **用户 Store** (`stores/user.store.ts`)：包含 `user`、`menus`、`permissions` 状态，以及 `login`、`logout`、`fetchUserInfo`
  方法。
- **应用 Store** (`stores/app.store.ts`)：管理 `sidebarCollapsed`、`isFullscreen` 及切换方法。
- **菜单组合式函数** (`composables/useMenu.ts`)：提供 `menus`（已过滤权限）和 `breadcrumbs`（基于当前路由生成）。
- **顶部栏组件** (`components/TopBar.vue`)：包含折叠按钮、面包屑、通知下拉、全屏切换、用户下拉。
- **用户下拉组件** (`components/UserDropdown.vue`)：显示用户信息，提供注销功能。
- **通知下拉组件** (`components/NotificationDropdown.vue`)：展示通知列表，支持标记已读。
- **默认布局** (`layouts/default.vue`)：左侧菜单（动态渲染，支持折叠）、右侧内容区使用 `TopBar` 和 `slot`。
- **登录页面** (`pages/login.vue`)：使用 `LoginForm` 组件，应用 `guest` 布局。
- **API 统一处理器** (`server/utils/defineApiEventHandler.ts`)：封装验证、守卫逻辑。
- **登录/注销/用户信息接口** (`server/api/auth/*.ts`)：提供认证相关 API。

## 九、其他注意事项

- 所有生成代码需遵循 **ESLint** 和 **Prettier** 默认配置（可假设已配置）。
- 组件内如需使用 Element Plus 图标，请从 `@element-plus/icons-vue` 导入具体图标组件，并在模板中使用 `<el-icon>` 包裹。
- 涉及路由跳转使用 `navigateTo`（Nuxt 3）而不是 `router.push`（但在组合式函数中可使用 `useRouter()`）。
- 确保生成的代码在 SSR 环境下兼容（避免直接使用浏览器 API 未做客户端判断）。
